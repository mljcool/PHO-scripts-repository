Executive Summary

What This Solution Is

A legacy Microsoft Dynamics CRM enterprise customization workspace for PHO / RICNAC. It contains server-side plugin DLLs, CRM form JavaScript web resources, SSRS report definitions, shared CRM helper libraries, test code, and supporting migration/configuration assets.

Business / System Purpose

The solution appears to support CRM operations around contacts, organizations/accounts, education, incidents or inquiries, campaigns, attendance, service requests, reporting, and historical migration data.

Main Technology Stack

Dynamics CRM SDK assemblies, Microsoft.Xrm.Sdk, C# .NET Framework class libraries, JavaScript Xrm form scripting, SSRS .rdl reports, SQL/Access, CSV/XML/Excel migration files, Visual Studio, MSBuild, and CRM Plugin Registration Tool.

High-Level Risks

Legacy CRM SDK references, .NET Framework dependencies, SSRS report project limitations, large migration assets, and missing plugin registration metadata that must be confirmed from the source environment.

02

Architecture Overview

The old solution is arranged around Dynamics CRM customizations. CRM records sit at the center; client scripts shape form behavior, server-side plugins enforce backend logic, reports consume CRM data, and migration/configuration assets support historical import, export, or deployment tasks.

High-Level CRM Flow

Server-Side Plugin Flow

  1. A CRM event such as create, update, assign, or send email is registered as a plugin step.
  2. The CRM runtime invokes the plugin assembly and passes IServiceProvider into Execute(...).
  3. The plugin reads IPluginExecutionContext, ITracingService, and IOrganizationService.
  4. The plugin validates the target record, performs business logic, and reads or updates CRM records.

Client-Side JavaScript Flow

  1. A CRM user opens or edits a record form.
  2. Entity-specific JavaScript web resources run on form load, field change, or save events.
  3. Shared libraries provide helper functions and service access wrappers.
  4. When the record is saved, server-side plugins may also execute in the CRM pipeline.

Reporting / Data Migration Flow

  1. RICNACReports contains SSRS .rdl files for operational reporting.
  2. RICNACFiles contains migration folders, Access databases, XML/CSV/Excel files, templates, site maps, and ZIP archives.
  3. Reports and migration files depend on entity schema, field names, relationships, and historical data conventions.
How Components Relate to CRM Records
Component Relationship to CRM Records Migration Consideration
Plugins Run when records are created, updated, or processed. Confirm target event registrations and decide whether plugins should be disabled during bulk migration.
JavaScript web resources Run on CRM forms before users save data. Review for deprecated Xrm.Page usage and form event compatibility.
Reports Read CRM data for operational output. Validate data source, query, and security changes in Dataverse.
Migration assets Represent historical/import data and supporting templates. Classify source data, map entities/fields, and confirm whether assets are still active.

Xrm Client API Usage Notes

  • Older files may use Xrm.Page or SOAP/OData helper patterns that require review for modern model-driven apps.
  • During migration, confirm each web resource is still referenced by forms, command bars, ribbons, or event handlers.
  • Shared libraries should be loaded before entity-specific scripts when forms depend on common helper functions.
  • For Dynamics 365 CE / Dataverse modernization, validate formContext usage, async API behavior, security role assumptions, and deprecated client methods.

Why .NET 7 / v7.1 Was Not Correct

  • TargetFrameworkVersion values such as v4.7.1 refer to .NET Framework targets.
  • v7.1 is not a valid .NET Framework target.
  • .NET 7 uses SDK-style projects with TargetFramework values such as net7.0.
  • Classic Dynamics CRM plugin assemblies are .NET Framework class libraries, not standalone .NET 7 apps.

CRM Compatibility Warnings

  • Local build success does not guarantee deployment compatibility with every CRM server version.
  • The target CRM/Dynamics 365 CE environment must support plugin assemblies compiled for .NET Framework 4.7.1.
  • The bundled Microsoft.Xrm.Sdk.dll is version 6.0.0.0, indicating an older CRM SDK generation.
  • Plugin behavior may depend on registration metadata not included in source files.

Reference Path Changes

  • CRM.Plugin.SequenceGenerator resolves Microsoft.Xrm.Sdk.dll from DigitalEmbrace_CRM_Solutions/RICNACLib/lib.
  • OAHPP.Plugins uses CrmSdkLibDir for Microsoft.Crm.Sdk.Proxy.dll and Microsoft.Xrm.Sdk.dll.
  • RICNACLib, RICNACPlugins, and RICNACTest already used local relative SDK references.

Remaining Technical Debt

  • Plugin registration exports are not present in the repository.
  • The full PHO solution does not build cleanly with plain MSBuild because of legacy report/file project entries.
  • Generated service references, old SDK DLLs, and legacy JavaScript toolkit files should be reviewed.
  • Migration assets may contain historical or sensitive data and should be classified before migration.
